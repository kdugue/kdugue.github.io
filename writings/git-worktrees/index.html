Scaling AI Development with Git Worktrees

April 25th, 2026

agents ai

Since transitioning to agent-based development, I’ve found myself wanting to extend my development workflow, especially with using multiple agents. I noticed a bottleneck when using agent workflows: If I had to work on multiple unrelated work items, I found myself having to stash changes and create another branch, finish work on that new branch, then come back to the original branch to finish my initial changes.

I recently discovered git worktrees and found that this was a perfect solution to my bottleneck.

At a conceptual level, git worktrees allow you to work on multiple branches of the same repository in separate directories simultaneously. In other words, you can work on different features or subfeatures while keeping the state of those changes intact (without needing to stash or make unmeaningful commits).

A real use case

Here’s a real use case of worktrees I did recently at work. I was working on a feature adding in a button component to an existing design. While working on that, data science requested updates for an event’s metadata.

My typical workflow would include:

  1. Stashing my current changes (or making an ‘in progress’ commit). Even in an agentic workflow, this is basically asking the agent to undo its changes. 

    git stash save "work in progress button"

  2. Create a new branch off main for the data change. Make those changes, put up a diff. 

    git checkout main
git checkout -b ds-metadata-update
# ... make changes ...
git commit -m "update metadata"
git push

  3. Come back to the branch for the button changes, revert back to my current working state by popping the stash. 

    git checkout feature-button
git stash pop

  4. Finish those changes and put up a second diff.

With git worktrees, the workflow looks like this:

  1. Create a worktree for the metadata changes.

    git worktree add ../metadata-update ds-metadata-update

  2. Make those changes in the new directory and put up a diff.

    cd ../metadata-update
# ... make changes ...
git commit -m "update metadata"
git push

  3. Go back to my original changes, finish up the button and put up a diff. 

    cd ../original-project-folder
# continue working immediately

In this new flow, I don’t have to worry about stashing and intentionally having to save my working state.

Why this matters for Agents

In an agentic workflow, worktrees are even more powerful. It allows for multiple agents to be working on different things since they can work in their own separate directories.

Without worktrees, you’re bottlenecked on making the change since checking out to a new branch means you need to stop work on the current branch. You wouldn’t want an agent to do that because:

  • Context Management: Switching branches within the same folder can quickly max out your context.
  • Separation of Concerns: There should be a physical separation of concerns for agents to ensure they aren’t bleeding logic across unrelated tasks.

By using worktrees, I’ve found I can scale my development alongside my agents. It allows for true parallel development, risk-free experimentation in separate “sandboxes,” and the ability to juggle multiple high-priority features without the overhead of keeping track of my changes.
